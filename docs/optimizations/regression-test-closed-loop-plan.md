# 项目级回归测试闭环实施计划

## 1. 背景与目标

Argus 已具备黑盒测试、白盒分析、任务报告、项目管理和 Web 控制台等单次任务能力。下一阶段
应将这些能力组织为可重复执行、可比较、可判定的项目级回归工作流，而不是继续进行没有明确
测量目标的全仓优化。

首期目标是让用户可以在一个项目内维护回归用例，手动或通过 API/CLI 批量执行，并相对于已选
基线判定本次是否可通过发布质量门禁。

### 成功标准

- 项目内的黑盒和白盒任务可保存为可复用的回归用例。
- 一次回归运行会创建独立、可审计的任务快照，不受后续用例编辑影响。
- 用户可以从控制台、REST API 或 CLI 发起和查看回归批次。
- 用户可显式选择一个成功批次作为项目基线。
- 任务失败、超时、取消，或相较基线新增高/严重问题时，批次被判定失败。
- 批次结果、差异和最终状态持久化；WebSocket 只承担实时通知而非事实来源。

## 2. 范围与非目标

### 本期范围

1. 项目级回归用例的创建、编辑、启用/禁用、排序和删除。
2. 回归批次的创建、任务快照、顺序提交、进度汇总和最终判定。
3. 项目基线的显式设置与读取。
4. 当前批次相对基线的问题差异计算和汇总报告。
5. 控制台、REST API 和 CLI 三个触发入口。
6. 持久化的批次查询与实时 WebSocket 进度事件。

### 非目标

- 定时调度、Cron、Webhook 触发和 CI 平台专用插件。
- 多进程、多副本、分布式 worker 或队列替换。
- 跨项目共享用例、用例参数矩阵、并行度配置。
- 可配置质量门禁阈值；首版使用固定规则。
- 改变现有单任务 HTML/JSON 报告格式、Python↔Java 分析器 HTTP 契约或既有任务 API 语义。

## 3. 领域模型与数据流

### 3.1 回归用例（RegressionCase）

回归用例归属于一个项目，保存以下可重放输入：

- 名称、任务类型（`blackbox` / `whitebox`）、目标、起始 URL。
- 现有任务参数、最大步骤、超时、截图开关与模型选择等可执行配置。
- `enabled`、显示顺序、创建/更新时间。

运行批次时，将用例配置复制到新建任务；任务保存批次和用例关联标识。任务仍是执行状态、
步骤、问题、产物和报告的唯一事实来源，不能让批次直接写入任务状态。

### 3.2 回归批次（RegressionRun）

批次记录项目、触发来源（控制台/API/CLI）、触发者、创建/开始/结束时间、状态、关联基线、
汇总信息和最终质量门禁结果。批次项记录用例快照、排序、子任务标识、执行状态和比较结果。

批次状态为：`pending`、`running`、`completed`、`failed`、`cancelled`。创建后由应用服务依次
创建并提交启用用例对应的子任务；全部子任务到达终态后再生成批次汇总和最终状态。

首版复用现有进程内队列和单 worker 约束，不新增并发模型。批次协调逻辑属于应用编排层，
共享存储、队列和事件发布器由 `RuntimeContainer` 装配。

### 3.3 基线与差异

仅允许成功批次被显式设为该项目的基线。新批次按相同回归用例关联到基线中的对应任务，
而不是按任务 ID 进行比较。

问题比较使用稳定指纹：

```text
task_type + finding_type + severity + normalize(title) + normalize(resource_location)
```

其中资源位置会去除易变行号；标题做空白、大小写等规范化。描述、生成时间、问题 ID、截图路径
不参与指纹，避免运行产物差异造成误报。差异分为新增、持续和已解决。

质量门禁固定规则如下：

- 任一子任务 `failed`、`timeout` 或 `cancelled`：批次失败。
- 相对基线新增 `high` 或 `critical` 问题：批次失败。
- 新增低/中问题、持续问题、已解决问题：写入差异报告，但不单独阻断。
- 没有基线时仅生成首份批次结果，不能产生回归差异；用户可在其成功后显式设为基线。

## 4. 接口与交互

### 4.1 REST API

在项目资源下提供回归用例 CRUD、创建/查询回归批次、查询批次报告及设置/读取项目基线的
兼容性新增接口。响应应携带批次状态、子任务摘要、基线引用、门禁结果和问题差异；所有新增
字段遵循现有 camelCase/OpenAPI 契约，并重新生成前端类型，禁止手改生成文件。

批次查询支持稳定分页和项目隔离。权限、项目归属校验、输入校验和错误响应复用现有 API
依赖与错误码，不在 route 中实现调度、SQL 或比较逻辑。

### 4.2 CLI

新增以下命令，且与 API 共用同一应用服务：

```bash
argus regression run --project <project-id>
argus regression status <run-id>
argus regression baseline set <run-id>
```

`run` 输出批次标识和可轮询状态；非通过批次应返回非零退出码，方便后续 CI 调用，但不在本期
实现任何特定 CI 平台集成。

### 4.3 Web 控制台

在项目详情增加“回归”页签：管理回归用例、启动批次、查看批次历史、设为基线和进入批次
差异报告。任务列表增加批次来源和关联跳转入口，但保留既有单任务详情与报告入口。

批次报告展示：通过/失败/跳过计数、阻断原因、每个用例当前状态、基线信息，以及新增、持续、
已解决问题。批次创建、用例开始/结束、批次完成使用现有 EventBus/WebSocket 实时发布；前端
重连后必须通过查询 API 恢复状态。

## 5. 持久化与兼容性

新增连续且幂等的 SQLite 版本化迁移，建立 `regression_cases`、`regression_runs`、
`regression_run_items`（或职责等价的关联表），并为项目、批次状态、批次项和任务关联建立必要索引。
如需在任务上保存批次/用例关联，使用可空兼容字段和外键；历史任务继续正常读取。

不得修改、改名或复用已发布迁移。现有项目、任务、报告、白盒 Java API 和文件产物路径保持兼容。
最终批次状态、子任务关联和差异结果写入 SQLite，保证服务重启后可查询；进程内 EventBus 历史
不能被当作可靠存储。

## 6. 分阶段实施与验收

### 阶段一：领域与持久化

- 实现领域模型、存储仓库、连续迁移和批次/基线状态机。
- 验收：用例配置能快照化；历史任务兼容；基线只能指向成功批次；迁移可重复执行。

### 阶段二：应用服务、API 与 CLI

- 实现批次协调服务、任务关联、汇总、差异和质量门禁；接入 REST、CLI、认证与事件发布。
- 验收：黑盒与白盒混合批次均可执行；失败规则准确；服务重启后可查询最终结果。

### 阶段三：控制台与批次报告

- 实现回归用例管理、批次历史/详情、基线操作和差异展示；同步 OpenAPI 类型。
- 验收：用户无需直接操作数据库即可完成“建用例 → 运行 → 查看差异 → 设基线”闭环。

### 阶段四：验证

- Python 单测：快照、状态机、汇总、基线、指纹、阻断/非阻断差异及取消/异常恢复。
- API 契约测试：鉴权、项目隔离、分页、非法状态转换和兼容响应。
- 前端测试：用例管理、批次启动、事件更新、差异展示和基线设置。
- 端到端场景：全部通过、任务失败、新增高危问题、新增低危问题、无基线、混合黑盒/白盒。

涉及 Java 代码时，除非用户明确要求，不执行 Maven 编译 / 测试，由用户自行验证。

## 7. 风险与后续演进

- 问题标题或位置不稳定可能造成误报：首版保留完整差异明细，收集真实样本后再调整规范化规则。
- 长批次在进程异常时无法恢复运行中的队列：首版以持久化终态查询和明确失败处理为准，不虚构
  分布式恢复能力。
- 基线可能被错误覆盖：仅允许显式设置成功批次，并在控制台显示确认信息和当前基线来源。

后续可在真实使用数据支持时评估：可配置质量阈值、用例参数矩阵、定时/CI 触发、批次并发控制、
外置队列与多副本一致性改造。任何扩容前必须遵守 `docs/architecture.md` 的单副本正确性、
契约稳定性和可测量收益要求。
